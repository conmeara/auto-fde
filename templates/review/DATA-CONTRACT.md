# review/data.js contract

`review.html` renders exclusively from `window.FDE_DATA`, defined in
`data.js` in the same directory. `data.js` is generated by
`scripts/gen-review-data.py` — never hand-edited. Tabs appear only when
their data section is present, so the page grows as the engagement advances.

```js
window.FDE_DATA = {
  generated: "2026-07-05T12:00:00Z",   // stamp from the generator
  engagement: {
    team: "Acme Launch Team",
    pluginName: "acme-launch",
    version: "0.1.0"
  },

  // ── Plan tab (present once catalog.json exists) ──────────────────────
  catalog: {
    phases: ["Anytime", "Planning", "Launch", "Wrap"],
    lanes: ["PM", "Comms"],            // optional, may be []
    skills: [{
      slug: "launch-brief",
      title: "Launch Brief",
      phase: "Planning",
      lane: "PM",                       // optional
      type: "document",                 // document | workflow | integration
      summary: "…",
      status: "planned",                // planned…deployed | cut
      composes: ["interview"],          // optional
      templates: ["templates/Launch-Brief.docx"],  // optional
      vendor: false,                    // optional
      notes: "…"                        // optional
    }],
    commands: [{ name, summary, status }],
    agents:   [{ name, summary, status }],
    integrations: [{ name, kind, authNotes }]
  },
  openQuestions: [{                     // optional; badge on Plan tab
    skill: "launch-brief",              // "" for engagement-level
    question: "…",
    status: "open"                      // open | resolved
  }],

  // ── Skills tab (present once skills are built) ───────────────────────
  builtSkills: {
    "launch-brief": {
      files: [{ path: "SKILL.md", content: "…" }],   // SKILL.md first,
                                                     // then references/, evals/
      scores: {                          // from verify; may be null
        fidelity: 9, bestPractices: 9, triggering: 9,
        verdict: "pass"                  // pass | revise | fail
      },
      openQuestions: ["…"]
    }
  },

  // ── Evals tab (present once /fde-eval has run) ───────────────────────
  evals: {
    trigger: {
      accuracy: 0.99, precision: 1.0, recall: 0.99,
      rounds: 2, totalQueries: 400,
      perSkill: [{ slug, tp, fn, tn, fp, failures: ["query …"] }],
      acceptedMisses: [{ slug, query, reason }]
    },
    output: { run: 3, passed: 3, gated: ["metrics-report"] },  // optional
    reportMarkdown: "…"                  // .build/eval-report.md contents
  },

  // ── Deploy tab (present once /fde-deploy starts) ─────────────────────
  deploy: {
    checklist: [{ item: "Marketplace manifest validated", done: true }],
    installSteps: ["…"],
    siteDir: "site/",
    notes: "…"
  }
};
```

Reviewer state (approve toggles, notes) lives in `localStorage`, keyed by
skill slug, with an Export button producing a markdown notes file `/fde-review`
can ingest. Rendering must tolerate missing optional fields and partial
localStorage state.
