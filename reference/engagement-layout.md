# Engagement layout

Every Auto-FDE engagement lives in one working directory (usually its own git repo, separate from this plugin). Every phase command reads and writes this layout — it is the handoff contract between phases. Check what exists here to know where the engagement stands.

```
<engagement>/
├── engagement.md            Engagement brief. Written by the kickoff interview in
│                            /fde-discover. Team, people, tools, materials, constraints.
├── discovery/
│   ├── sources/             Raw team materials (docs, templates, exports). Often
│   │                        confidential — gitignored by default.
│   ├── transcripts/         Meeting and interview transcripts awaiting ingestion.
│   └── digests/             Normalized knowledge digests (*.md). The ONLY discovery
│                            output later phases read. All three intake modes
│                            (corpus, transcript, interview) write this format.
├── catalog.json             The skill catalog manifest — single source of truth for
│                            what gets built. Written by /fde-plan, updated by every
│                            later phase (status field per item). Schema:
│                            ${CLAUDE_PLUGIN_ROOT}/templates/schemas/catalog.schema.json
├── dashboard.html           The one evolving engagement dashboard, at the root.
│                            Self-contained: ${CLAUDE_PLUGIN_ROOT}/scripts/gen-dashboard.py
│                            embeds the data (and copies the template in on first run);
│                            never hand-edit, regenerate. Published as a Claude artifact
│                            so the champion keeps one stable link. Home tab = state +
│                            next command + guided tour; Plan/Skills/Test/Evals/Deploy
│                            tabs appear as data exists.
├── .build/                  Machine outputs: workflow run records, verify scores,
│                            open-questions.json, approvals.json (durable gate
│                            records), test-log.json, trigger-report.json,
│                            output-evals.json, output-runs/ (per-run claude -p
│                            transcripts), practice-report.json, regressions.json,
│                            feedback-log.json, practice-run/. Gitignored.
├── <plugin-name>/           The team plugin being built (name from catalog.json).
│                            Standard plugin layout: .claude-plugin/plugin.json,
│                            commands/, agents/, skills/, templates/, reference/,
│                            scripts/run-checks.py (the deterministic eval checker,
│                            copied in at scaffold time).
├── test/                    Test-phase kits (test/kits/<journey>/): one synthetic
│                            starting artifact + README (opening prompt, watch-fors)
│                            per entry-point journey, for the human's live testing.
├── practice/                Synthetic end-to-end test project generated by /fde-eval.
├── site.html                Team-facing site (Overview / Install / Wiki), one
│                            self-contained page like the dashboard: generated by
│                            ${CLAUDE_PLUGIN_ROOT}/scripts/gen-site.py from catalog.json,
│                            .build/deploy.json, and .build/wiki/; published as a Claude
│                            artifact — one stable link for the whole team.
└── feedback/                Field misfire reports, one file per report. /fde-improve
                             converts them into eval cases (ledger:
                             .build/feedback-log.json).
```

## Phase → state map

| Phase command | Requires | Produces |
| --- | --- | --- |
| `/fde-discover` | nothing (kickoff) or `discovery/sources|transcripts` | `engagement.md`, `discovery/digests/*.md` |
| `/fde-plan` | digests | `catalog.json`, `dashboard.html` (Plan tab), `.build/open-questions.json` |
| `/fde-build` | `catalog.json` approved | `<plugin-name>/` skills+commands+agents, `.build/verify-scores.json`, dashboard Skills tab |
| `/fde-review` | built plugin | skill revisions, regression cases pinned in evals + `.build/regressions.json`; description edits re-run the trigger benchmark, body edits the output evals |
| `/fde-test` | reviewed plugin | `test/kits/`, `.build/test-log.json` (Test tab), real-phrasing eval cases + queued fixes from the user's live sessions |
| `/fde-eval` | tested plugin (or the user's go-ahead) | `.build/trigger-report.json` (incl. live calibration), `.build/output-evals.json` + `output-runs/`, `.build/practice-report.json` (Evals tab), `practice/` |
| `/fde-deploy` | evals passing | packaged marketplace, `site.html` (+ `.build/wiki/`, `.build/video-briefs/`), `.build/deploy.json` |
| `/fde-improve` | `feedback/` reports | eval cases + fixes, updated `.build/regressions.json`, refreshed dashboard |

## Rules

- **Digests are the only bridge out of discovery.** Synthesis never reads raw sources directly; if a digest is missing detail, fix the digest (re-extract or interview), don't work around it.
- **catalog.json is the only source of truth for scope.** The dashboard, build workflow, eval catalog, and site all derive from it. Never let a phase keep a private copy of the skill list.
- **Raw sources stay out of git and out of the built plugin.** Only blank templates explicitly listed in `catalog.json` get copied into `<plugin-name>/templates/` — by hand or scripted copy, never retyped by an agent.
- **Open questions accumulate; interviews resolve them.** Every phase may append to `.build/open-questions.json`; the discovery interview mode is the standard way to burn the list down.
- **Every fixed finding becomes an eval case.** Review fixes, test-drive failures, practice failures, and field reports are pinned in the owning skill's evals and tracked in `.build/regressions.json` — the benchmark, not the fixer, flips them to passing.
- **Gate approvals are recorded, not remembered.** Each gate decision appends `{gate, date, by, note}` to `.build/approvals.json` — chat memory doesn't survive sessions, and "the user approved it earlier" without a record is not an approval.
- **"The champion" is the team-side owner of the engagement.** When someone implements Claude for their own team, operator and champion are the same person — every phase works identically; the gates just become their own checkpoints.
