# Auto-FDE v1 plan

Auto-FDE turns the FDE lifecycle — discover a team's workflows, plan a skill
catalog, build a Claude plugin, review it, eval it, deploy it — into a
reusable Claude Code plugin. The methodology was proven once by hand (the
creative-studio engagement); v1 generalizes that run.

## Decisions (locked 2026-07-05)

- **Claude Code only.** The Workflow tool, preview tools, and plugin CLI are
  load-bearing; no multi-host support in v1.
- **Users:** team champions implementing Claude for their own team, and
  FDEs/consultants doing it for client teams. Maximum automation, human at
  review gates. The plugin also *teaches* the FDE method as it runs it.
- **Command prefix `/fde-*`** — `/fde` dashboard plus one command per phase.
- **One evolving review page.** A single `review/review.html` per engagement
  gains tabs as phases complete: Plan → Skills → Evals → Deploy. One design
  system, one generated `data.js`.
- **Fully generic templates.** The repo is public. All shipped templates and
  examples use the fictional **Acme Launch Team**. No real team content.
- **v1 done bar: structural validation.** plugin-validator-style checks pass
  and the trigger-eval workflow runs clean on the example catalog. (A full
  synthetic dogfood run is the first post-v1 milestone.)
- **Source strategy:** Anthropic sources (plugin-dev, skill-creator, Complete
  Guide) are the backbone — distilled closely with attribution. Third-party
  sources (Pocock weighted most, Ondrej, Steinberger's
  conciseness/anti-duplication points) are distilled as ideas, never
  vendored. All three distillates live inside the doctrine skill at
  `skills/skill-authoring/references/` (single source of truth) and feed its
  SKILL.md compass.

## Architecture

State backbone: every engagement is a working directory with a fixed layout
([reference/engagement-layout.md](reference/engagement-layout.md)). Two
artifacts carry all handoffs: **knowledge digests** (the only output of
discovery, regardless of intake mode) and **catalog.json** (the only source
of truth for scope; schema in `templates/schemas/`).

```
commands/   fde (dashboard) · fde-discover · fde-plan · fde-build ·
            fde-review · fde-test · fde-eval · fde-deploy · fde-improve
                                                     — thin state-aware routers
skills/     skill-authoring (the doctrine/compass) + one method skill per phase:
            discovery · planning · building · reviewing · testing ·
            evaluating · deploying · improving
agents/     plugin-validator — structural validation gate (original, checklist
            distilled from plugin-dev)
scripts/    extract-corpus.workflow.js   — fan-out digest extraction
            build-skills.workflow.js     — build → verify → revise per skill
            revise-skills.workflow.js    — review-note-driven revision pass
            eval-trigger.workflow.js     — trigger benchmark (hold-out split,
                                           blind judges)
            eval-output.workflow.js      — output evals (3× runs + no-skill
                                           baseline, isolated worktrees)
            practice-run.workflow.js     — execute + grade the practice project
            gen-dashboard.py             — scan engagement → dashboard.html
            run-checks.py                — deterministic checks.json runner
                                           (copied into every built plugin)
            dev-sync.sh                  — uninstall+reinstall install loop
templates/  dashboard/ — the evolving dashboard (one self-contained page)
            site/      — site.html, the team-facing artifact page
                         (Overview / Install / Wiki), generated by gen-site.py
            schemas/   — catalog schema+example, digest + engagement-brief templates
reference/  engagement-layout.md — the state convention (loaded by every phase)
references/ upstream sources (repo docs, not shipped logic)
```

### Discovery: three intake modes, one output

Corpus extraction (folder of docs → fan-out workflow), transcript ingestion
(meetings/interviews → same digest format), and live interview (Claude
elicits from the champion/SMEs; also the standard mechanism for burning down
open questions from any phase). Kickoff interview produces `engagement.md`
before any extraction.

### Build: the proven pipeline

Per catalog skill: build (from digests + real sources) → adversarial verify
(scored fidelity / best-practices / triggering + verdict) → revise if not
pass. Doctrine injected from the `skill-authoring` skill. Validator gate at
the end. Lessons encoded: extract once and build from extracts; never
bottleneck a fan-out on one giant structured output; edit the script's slug
filter for pilots instead of relying on `args` over `scriptPath`; templates
get bundled and wired, never described.

### Eval: three rungs

Trigger evals (automatable benchmark, target ≥0.99 accuracy / 1.0 precision),
output evals (gated on live connections), generated practice project
(synthetic client + runbook for end-to-end testing).

## v2 — eval flywheel + dashboard (2026-07-07)

Locked decisions, applied on top of v1:

- **`dashboard.html` at the engagement root replaces `review/`.** One
  self-contained file generated by `scripts/gen-dashboard.py` (copies the
  template in on first run, embeds the data) and published as a Claude
  artifact — the same path redeploys to the same URL, so the champion keeps
  one stable link. The Home tab is the operating surface: where the
  engagement stands, the exact next command, gates waiting, attention items,
  and an 8-step lifecycle spine that doubles as a guided tour on fresh
  engagements. Plan/Skills/Evals/Deploy tabs appear as data exists.
- **Four eval artifacts per built skill:** `evals/trigger-evals.json`,
  `evals/evals.json` (a `gated` field names any live connection an eval
  needs — gated evals are listed, never mocked), `evals/checks.json`
  (deterministic graders — contains / contains-all / regex / path-resolves;
  `judge` only for what code can't check), and `evals/reference/<evalId>.md`
  — a reference solution per output eval that passes every check.
  catalog.json skill entries gain a `success` list written at plan time
  (required by the method for every buildable skill; schema-optional so
  vendor/legacy entries stay valid); build compiles it into checks.json.
- **Trigger benchmark v2** (`eval-trigger.workflow.js`): deterministic 60/40
  train/hold-out split, three blind judges per skill, script-side majority
  scoring. The hold-out score is the headline (target ≥0.99 accuracy at
  precision 1.0); descriptions are fixed from train failures only.
  Relabeling a case or accepting a miss requires a replacement case — the
  set only grows. Report: `.build/trigger-report.json`.
- **Output evals with baselines** (`eval-output.workflow.js`): each runnable
  eval runs with-skill 3× plus a no-skill baseline arm, in isolated
  worktrees; deterministic checks graded first, rubric judge second. A case
  passes only if every run passed; a passing baseline flags the skill as a
  cut candidate. Results: `.build/output-evals.json`.
- **Practice run executed, not just generated**
  (`practice-run.workflow.js`): an agent plays the champion and drives the
  runbook sequentially in `.build/practice-run/work/`; each step is graded
  with the owning skill's checks; failures file eval cases. Results:
  `.build/practice-report.json`; the graded run seeds the deploy site's
  tutorials.
- **Regression ledger:** every fixed finding (review note, trigger triage,
  practice failure, field report) is pinned as a permanent eval case and
  recorded in `.build/regressions.json`. Any description edit forces a
  trigger-benchmark re-run before the phase reports done.
- **Misfire channel + `/fde-improve`:** teammates report wrong-skill-fired /
  bad-output moments in one message; reports land as files in
  `<engagement>/feedback/`. The new `improving` skill converts each report
  into an eval case, reproduces, fixes, re-runs both benchmarks, updates
  `.build/feedback-log.json` and `regressions.json`, regenerates +
  republishes the dashboard, and folds recurring lessons into the doctrine
  reference.

## v3 — the copilot correction (2026-07-07)

Framing: Auto-FDE is a **copilot** that walks its user through the
engagement — anyone runs the same process. Automation owns mechanical work
(checks, validation, benchmarks, generation); the user owns judgment work
(testing, triage calls, approvals); reviewers stay at the gates. Branding
and pitch unchanged. Driven by the 2026-07-07 audit against skill-creator,
demystifying-evals, OpenAI eval-skills, the CC skills docs, and plugin-dev
(14 confirmed findings).

**The lifecycle gains a Test phase — 9 steps:**
Discover → Plan → Build → Review (read + revise on the Skills tab) →
**Test** (use it live, a lot) → Eval (formalize what testing found) →
Deploy → Improve. Test and Eval stay separate on purpose: Test is the
human learning whether the plugin feels right and harvesting cases;
Eval is the machine proving routing and outputs hold with hold-outs and
baselines — the dogfooding-vs-evals division the guidance draws.

### WP1 — Checks are code

`scripts/run-checks.py` executes `contains` / `contains-all` / `regex` /
`path-resolves` deterministically with per-check evidence (matched or
missing value, which file); copied into every built plugin's `scripts/`
at scaffold time so the team's evals stay runnable after handoff. The
eval-output grader, practice-run grader, and build verifier shell out to
it and copy its results verbatim; agents rule only on `judge`-kind checks
and the rubric.

### WP2 — Measure reality, not simulations

- **Output evals on the real harness:** each arm is a headless `claude -p`
  run in its isolated worktree — with-skill loads the plugin via
  `--plugin-dir`, baseline runs bare; **both arms 3×**. Session
  transcripts persist to `.build/output-runs/` (no more self-reported
  summaries); `tokens`/`seconds` fill the fields the data contract
  reserves; new trace-side check kinds (`used-tool`, `ran-command`) make
  process criteria decidable. Every run's verdict is reported, not just
  run 1's. Practice-run steps run the same way.
- **Live trigger calibration:** judges stay the cheap iteration loop, but
  before the phase reports done, a script-selected hold-out sample (~30
  queries no human or agent reads) runs via `claude -p` in fresh sessions
  with the plugin installed, 3× each, counting real Skill-tool
  invocations. `trigger-report.json` and the dashboard carry the
  judge-vs-live delta.
- **Listing-budget gate:** total always-on description size checked
  against the ~1%-of-context skill-listing budget (validator error when
  over); `/doctor` check on the pilot install; planning levers for
  40+-skill catalogs (name-only low-priority skills,
  `skillListingBudgetFraction` note in TESTING.md).

### WP3 — The Test phase (`/fde-test` + testing skill + Test tab)

Solves "it's hard to start testing" and seeds the eval flywheel from real
usage:

- **Kits:** per entry-point journey, a ready-made kit — a synthetic
  starting artifact (meeting transcript, request doc; shares the practice
  fixture's fictional client), the opening prompt, and what to watch for
  — plus the `claude --plugin-dir` one-liner to load the plugin live.
- **Ingest:** `/fde-test` accepts the test session's transcript (path or
  paste). Mining it yields: skills exercised (coverage), trigger
  hits/misses, the user's inline corrections ("missed the tone of voice
  here") → revision notes routed through the review machinery, and
  confirmed behaviors → pinned eval cases in the user's real phrasing
  (`source: "test drive"`).
- **Tracking:** `.build/test-log.json` — series → sessions →
  {skillsExercised, verdicts, notes, transcript, casesFiled}. The
  dashboard Test tab shows a per-skill coverage grid, series notes, and
  an untested-skills nudge.
- **Expansion → autonomous runs:** after test series, generate variation
  cases anchored to observed sessions; the user skims the list, then the
  eval workflows run everything autonomously. **Provenance rule:**
  human-observed cases are eligible for hold-out; `synthetic-expansion`
  cases score as train only — the headline never inflates on generated
  cases. (Also retires the pre-deploy half of v2's same-mind authorship
  limitation.)

### WP4 — Plugin-dev completeness

- `claude plugin validate --strict` is the first, blocking step of the
  build and deploy gates, and lands in TESTING.md + dev-sync.sh; the
  validator agent keeps what the CLI can't do (leaked paths, confidential
  terms, eval artifacts, listing budget).
- Hooks and MCP become first-class: catalog.schema.json gains optional
  `hooks` entries (event, matcher, type, purpose) and per-integration
  `mcp` config (server type, command/url, env vars); planning asks the
  questions; building scaffolds `hooks/hooks.json` and `.mcp.json`.
- Built plugins get a generated README.md (from catalog.json) and
  .gitignore always; LICENSE when publishing beyond the team.
- `agents/plugin-validator.md` gets the conventional `model`/`color`
  frontmatter; the scaffold points at the agent-frontmatter table.
- `references/anthropic-plugin-dev.md` refreshed against live docs
  (custom manifest paths REPLACE defaults for commands/agents;
  `when_to_use` supported; `/reload-plugins` covers hooks; newer
  frontmatter and agent fields) with a re-fetch cadence in its header.

### WP5 — Deploy as guided rollout

- **Approvals walkthrough:** per integration, the exact connection steps
  (MCP add, `/mcp` auth, tokens) with the copilot waiting then verifying;
  marketplace path: validate --strict → marketplace add → install →
  fresh-session smoke tests, restarts explicitly the user's step.
  Checklist recorded in `.build/deploy.json` → Deploy tab.
- **Site becomes a Claude artifact:** `templates/site/` is replaced by
  one self-contained `site.html` at the engagement root (data markers +
  `gen-site.py`, tabs Overview / Install / Wiki), published via the
  Artifact tool — one stable team link, no hosting step.
- **Wiki via workflow:** `gen-wiki.workflow.js` — one author per phase
  router writing a tutorial from real test/practice transcripts in the
  team's vocabulary, a verify pass checking every command and path
  against the plugin.
- **Videos optional via ripple + ui-backlot** (separate plugins):
  `.build/video-briefs/<slug>.md` per demo slot (the graded transcript is
  the script; ui-backlot for app-UI surfaces, ripple for edit/QA);
  invoked when those plugins are installed, otherwise shipped as a
  handoff list. Site slots render placeholders until files exist.

### Cross-cutting

Transcript-first triage (before filing any fix from a failed case, read
the failing run's transcript: agent mistake vs grader defect); capability
evals (cut/deferred catalog entries pinned as expected-fail — the hill),
model-upgrade re-run trigger and saturation flag in `/fde-improve`;
review-phase symmetry (body edits after output evals exist force an
output re-run; `staleOutput` chip); durable gate records in
`.build/approvals.json`; single-operator voice ("you") across all docs;
human-facing warning to never open the hold-out labels in
`.build/trigger-cases.json`.

**Sequencing:** WP1 → WP2 → WP3 → WP4 → WP5; each WP closes with data
contract + dashboard updates, a dogfood run, and this file.

**Implemented in full 2026-07-07** (same day as the audit). Verification
highlights: run-checks.py fixture-tested across all seven kinds including
trace checks against a real `claude -p` transcript; the live-harness
mechanic proven end to end (Skill tool observed firing under
`--plugin-dir`); dashboard + site pages render zero-error on example,
fixture, and null data; `claude plugin validate --strict` passes clean on
this repo. Dogfood round 3 (9 skills, 177 queries): judge hold-out
1.000/1.000/1.000, agreement 1.000 — and the FIRST live calibration
measured **0.625 live accuracy at 1.0 precision (judges +37.5%
optimistic)** on an 8-query sample, confirming the audit's central claim
with numbers; interpretation and discipline notes in
`.build/eval-report.md`. The pre-deploy corrective for that gap is
exactly the new Test phase.
